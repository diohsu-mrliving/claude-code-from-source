# 第九章：Fork 代理與提示詞快取

## 九十五趴洞見

當一個父代理同時派生五個子代理時，每個子代理的 API 請求中，絕大多數內容都是相同的。系統提示詞相同、工具定義相同、對話歷史相同、觸發派生的助理訊息也相同。唯一不同的，是最後那一行指令：「你負責資料庫遷移」、「你來寫測試」、「你去更新文件」。

在一次典型的 fork，且對話已有一定深度的情況下，共用的前綴可能多達 80,000 個 token。每個子代理專屬的指令可能只有 200 個 token。這代表有 99.75% 的重疊。Anthropic 的提示詞快取對已快取的輸入 token 提供 90% 的折扣。如果你能讓這 80,000 個 token 在第二到第五個子代理時命中快取，那四個請求的輸入成本就直接砍掉九成。對父代理來說，這是花 $4 還是花 $0.50 的差別——完成同一批平行任務。

麻煩的是，提示詞快取必須精確到位元組層級。不是「夠像就好」，也不是「語意上等價」。從系統提示詞第一個位元組開始，到各子代理內容開始分歧之前的最後一個位元組，每一個字元都必須完全吻合。多一個空格、工具定義的順序換了、某個功能旗標讓系統提示詞的一個片段改變——快取就失效了。整個前綴以全價重新處理。

Fork 代理是 Claude Code 應對這個限制的解法。它不只是「帶著上下文產生子代理」的便利功能——它是一個偽裝成協調功能的提示詞快取利用機制。Fork 系統的每一個設計決策，都源自同一個問題：我們如何保證平行子代理之間的前綴在位元組層級完全相同？

---

## Fork 子代理繼承什麼

一個 fork 代理會從父代理繼承四樣東西，且繼承方式是傳參考或位元組精確複製，而非重新計算。

**1. 系統提示詞。** 不是重新生成，而是直接傳遞。父代理已渲染完成的系統提示詞位元組，透過 `override.systemPrompt` 傳入，來源是 `toolUseContext.renderedSystemPrompt`。這正是父代理在最近一次 API 呼叫中送出的那個字串。

**2. 工具定義。** Fork 代理定義宣告 `tools: ['*']`，但設定了 `useExactTools` 旗標為 true，子代理會直接收到父代理組裝好的工具陣列。沒有過濾、沒有重新排序、沒有重新序列化。

**3. 對話歷史。** 父代理與 API 之間的每一則訊息——使用者回合、助理回合、工具呼叫、工具結果——都透過 `forkContextMessages` 克隆到子代理的上下文中。

**4. 思考設定與模型。** Fork 定義指定 `model: 'inherit'`，解析結果為父代理的確切模型。相同模型代表相同的分詞器、相同的上下文視窗、相同的快取命名空間。

Fork 代理定義本身極簡，幾乎等同於空操作：

Fork 代理定義刻意保持極簡——它從父代理繼承一切。它指定所有工具（`'*'`），繼承父代理的模型，使用 bubble 模式處理權限（讓提示浮現在父代理的終端機），並提供一個從不實際被呼叫的空系統提示詞函式——真正的提示詞透過 override 頻道送達，已渲染完成且位元組穩定。

## 位元組相同的前綴技巧

送給 Claude 的 API 請求有固定結構：系統提示詞，然後是工具，然後是訊息。要讓提示詞快取命中，從請求開頭到某個前綴邊界為止的每個位元組，必須在所有請求之間完全相同。

Fork 代理透過確保三個層次凍結來達成這一點：

**第一層：透過傳遞而非重新計算來傳遞系統提示詞。**

當父代理的系統提示詞在最後一次 API 呼叫時完成渲染，結果就被捕獲在 `toolUseContext.renderedSystemPrompt` 裡。這是所有動態插值完成後的字串——GrowthBook 功能旗標、環境細節、MCP 伺服器描述、技能內容、CLAUDE.md 檔案。Fork 子代理收到的正是這個確切字串。

為什麼不直接再呼叫一次 `getSystemPrompt()`？因為系統提示詞的生成並非純函式。GrowthBook 旗標會從冷態過渡到熱態，因為 SDK 會去拉取遠端設定。一個在父代理第一個回合返回 `false` 的旗標，等到 fork 子代理啟動時可能已經返回 `true`。如果系統提示詞包含了由這個旗標控制的條件區塊，重新渲染的提示詞就算只差一個字元也會分歧。快取失效，80,000 個 token 全價重算，乘以五個子代理。

傳遞已渲染的位元組，消除了這整類分歧問題。

**第二層：透過精確傳遞來傳遞工具定義。**

一般的子代理會走過 `resolveAgentTools()`，依據代理定義的 `tools` 和 `disallowedTools` 陣列過濾工具池、套用權限模式差異，並可能重新排序工具。序列化後的工具陣列會和父代理不同——子集不同、順序不同、權限標注不同。

Fork 代理完全跳過這一步：

```typescript
const resolvedTools = useExactTools
  ? availableTools  // 父代理的確切陣列
  : resolveAgentTools(agentDefinition, availableTools, isAsync).resolvedTools
```

`useExactTools` 旗標只在 fork 路徑設為 true。子代理照單全收父代理的工具池。工具相同、順序相同、序列化相同。這包含保留 Agent 工具在子代理的工具池中，即便子代理被禁止使用它——移除它會改變工具陣列，破壞快取。

**第三層：訊息陣列的構造。**

這裡是 `buildForkedMessages()` 細心工作的地方。這個函式構造最後兩則訊息，放置在共用歷史和各子代理指令之間：

`buildForkedMessages()` 函式構造最後兩則訊息，位於共用歷史與各子代理指令之間。演算法如下：

1. 克隆父代理的助理訊息（保留所有 `tool_use` 區塊及其原始 ID）。
2. 對每個 `tool_use` 區塊，建立一個使用常數佔位字串的 `tool_result`（所有子代理相同）。
3. 建構一則使用者訊息，包含所有佔位結果，再加上包在樣板標籤裡的各子代理指令。
4. 返回 `[clonedAssistantMessage, userMessageWithPlaceholdersAndDirective]`。

```typescript
// 虛擬碼——示意訊息構造方式
function buildChildMessages(directive, parentAssistant) {
  const cloned = cloneMessage(parentAssistant)
  const placeholders = parentAssistant.toolUseBlocks.map(b =>
    toolResult(b.id, CONSTANT_PLACEHOLDER)  // 所有子代理均相同的位元組
  )
  const userMsg = createUserMessage([...placeholders, wrapDirective(directive)])
  return [cloned, userMsg]
}
```

每個子代理的最終訊息陣列看起來像這樣：

```
[...shared_history, assistant(all_tool_uses), user(placeholder_results..., directive)]
```

在指令之前的每個元素，在所有子代理之間都是相同的。`FORK_PLACEHOLDER_RESULT`——一個常數字串 `'Fork started -- processing in background'`——確保就連工具結果區塊也是位元組相同的。`tool_use_id` 值相同，因為它們參照同一則助理訊息。只有最後那個包含各子代理指令的文字區塊會有所不同。

快取邊界就落在那個最終文字區塊之前。在此之前的一切——可能多達數萬個 token 的系統提示詞、工具定義、對話歷史和佔位結果——對第一個子代理之後的每個子代理都以 90% 的折扣命中快取。

## Fork 樣板標籤

每個子代理的指令被包在一個樣板 XML 標籤中，它有兩個用途：指導子代理的行為，以及作為遞迴 fork 偵測的標記。

樣板包含大約 10 條規則。關鍵規則如下：

- **覆寫父代理的 fork 指令。** 父代理的系統提示詞說「預設使用 fork」——樣板明確告訴子代理：「那條指令是給父代理的。你就是那個 fork。不要產生子代理。」
- **靜默執行，最後報告一次。** 工具呼叫之間不要有對話文字。直接使用工具，然後產生一份結構化摘要。
- **保持在範圍內。** 子代理不能超出其指令範圍。
- **結構化輸出格式。** 回應必須遵循「範疇/結果/關鍵檔案/已修改檔案/問題」的範本，讓父代理在多個子代理同時回報時易於解析。

第一條規則特別值得關注。父代理的系統提示詞——子代理基於快取考量原封不動繼承的那份——包含像「有平行工作時預設使用 fork」這樣的指令。如果子代理遵循這條指令，它會嘗試 fork 自己的子代理，製造無限遞迴的代理。樣板明確覆寫：「那條指令是給父代理的。你就是那個 fork。」

結構化輸出格式（範疇/結果/關鍵檔案/已修改檔案/問題）並非裝飾性的。它把子代理的輸出限制在事實陳報，讓父代理在五個子代理同時回報時更易於解析和彙整。

## 遞迴 Fork 預防

Fork 子代理保留 Agent 工具在其工具池中。這是必須的——移除它會改變序列化的工具陣列，破壞提示詞快取。但如果子代理真的呼叫 Agent 工具且不指定 `subagent_type`，fork 路徑就會再次觸發，製造一個孫代理 fork。這個孫代理會繼承更大的上下文（父代理加子代理的對話），產生自己的 fork，如此無限遞迴下去。

兩道防護措施阻止這種情況：

**主要防護：querySource 檢查。** 當一個 fork 子代理被派生時，其 `context.options.querySource` 被設為 `'agent:builtin:fork'`。`call()` 方法在允許 fork 路徑之前會先檢查這一點：

```typescript
// 在 AgentTool.call() 中：
if (effectiveType === undefined) {
  // Fork 路徑——但我們已經在 fork 中了嗎？
  if (querySource === 'agent:builtin:fork') {
    // 拒絕：已經是 fork 子代理
  }
}
```

這是快速路徑。它只需檢查 options 物件中的一個字串。

**備用防護：訊息掃描。** Fork 預防使用兩道防護：在派生時設置的 `querySource` 標籤（快速路徑——單一字串比較），以及掃描訊息歷史中的樣板 XML 標籤的備用方案。備用防護的存在，是因為 `querySource` 在 autocompact 後仍會保存，但在邊緣情況下若未被正確傳遞，訊息掃描備用方案就能捕捉到遞迴。這是一種防護層疊的方法，其中檢查的成本（掃描訊息）相比於意外遞迴 fork 的代價（失控的 API 支出）微不足道。

為什麼要有備用防護？因為 Claude Code 有一個 autocompact 功能，當上下文過長時會重寫訊息陣列。Autocompact 可以重寫訊息內容，但會保留 options 中的 `querySource`。理論上，`querySource` 單獨就足夠了。但實際上，訊息掃描備用防護能捕捉 `querySource` 未被正確傳遞的邊緣情況——這是一種防護層疊方法，其中檢查的成本（掃描訊息）相比於意外遞迴 fork 的代價（失控的 API 支出）微不足道。

## 同步到非同步的轉換

一個 fork 子代理一開始在前台執行：其訊息串流到父代理的終端機，父代理阻塞等待完成。但如果子代理花了太長時間怎麼辦？Claude Code 允許在執行中途將代理推移到背景——使用者（或自動逾時機制）可以把一個正在前台執行的代理推移到背景，且不會遺失任何工作。

這個機制出乎意料地簡潔：

1. 當一個前台代理透過 `registerAgentForeground()` 註冊時，會建立一個背景化信號 Promise。

2. 父代理的同步迴圈在代理訊息流和背景化信號之間競速：

```
while (true) {
  const result = await Promise.race([
    iterator.next(),         // 代理的下一則訊息
    backgroundSignal,        // 「移至背景」觸發器
  ])
  if (result === BACKGROUND_SIGNAL) break
  // ... 處理訊息
}
```

3. 當背景化信號觸發，前台迭代器透過 `iterator.return()` 優雅地終止。這觸發生成器的 `finally` 區塊，處理清理工作。

4. 一個新的 `runAgent()` 實例以 `isAsync: true` 派生，使用相同的代理 ID 和到目前為止累積的訊息歷史。代理從中斷的地方繼續，現在在背景中執行。

5. 原本的同步 `call()` 返回 `{ status: 'async_launched' }`，父代理繼續其對話。

沒有任何工作遺失，因為訊息歷史就是代理的狀態。磁碟上的旁鏈記錄包含代理產生的每一則訊息。新的非同步實例從這份記錄重播，並從同步實例停下的地方繼續。

## 自動背景化

當 `CLAUDE_AUTO_BACKGROUND_TASKS` 環境變數或 `tengu_auto_background_agents` GrowthBook 旗標啟用時，前台代理會在 120 秒後自動背景化：

透過環境變數或功能旗標啟用後，前台代理會在 120 秒後自動背景化。停用時，函式返回 0（不自動背景化）。

這是一個具有成本影響的 UX 決策。前台代理會阻塞父代理的終端機——使用者無法輸入、無法下新指令、無法派生其他代理。兩分鐘對大多數快速任務來說足夠代理同步完成（此時串流輸出是有用的回饋），但對於長時間執行的任務來說又不至於讓終端機被佔用太久。

在 fork 實驗模式下，自動背景化的問題根本不存在：所有 fork 派生從一開始就強制為非同步。`run_in_background` 參數從 schema 中完全隱藏。每個 fork 子代理在背景中執行，完成後透過 `<task-notification>` 回報，父代理永遠不會阻塞。

## 何時不使用 Fork

Fork 是幾種協調模式之一，且刻意在三種情況下排除：

**協調者模式（Coordinator mode）。** 協調者模式和 fork 模式是互斥的。協調者有一個結構化的委派模型：維護計畫、以明確提示詞將任務分配給工作者、追蹤進度。Fork 的「繼承一切」方式會破壞這一點。一個被 fork 的協調者會繼承父協調者的系統提示詞（上面寫著「你是協調者，委派工作」），子代理就會試圖去協調而非執行。`isForkSubagentEnabled()` 函式會先檢查 `isCoordinatorMode()`，若為活躍狀態則返回 false。

**非互動式會話。** SDK 和 API 消費者（`--print` 模式、Claude Agent SDK）在沒有終端機的情況下運作。Fork 的 `permissionMode: 'bubble'` 會把權限提示浮現到父代理的終端機——而在非互動模式中，終端機並不存在。與其建立一套獨立的權限流程，fork 路徑直接被停用。SDK 消費者改用明確的 `subagent_type` 選擇。

**明確的 subagent_type。** 當模型指定了 `subagent_type`（例如 `"Explore"`、`"Plan"`、`"general-purpose"`），fork 路徑不會觸發。Fork 只在 `subagent_type` 省略時觸發。這讓模型可以在「我想要一個有自己系統提示詞和工具集的專用代理」（明確指定類型）和「我想要一個繼承我上下文的克隆體來平行處理這件事」（省略類型）之間做選擇。

## 經濟學

考慮一個具體場景。開發者請 Claude Code 重構一個模組。父代理分析程式庫，制定計畫，並同時派發五個 fork 子代理：一個更新資料庫 schema，一個重寫服務層，一個更新路由器，一個修復測試，一個更新型別定義。

在對話的這個時間點，共用上下文相當可觀：
- 系統提示詞：約 4,000 個 token
- 工具定義（40+ 個工具）：約 12,000 個 token
- 對話歷史（分析 + 規劃）：約 30,000 個 token
- 包含五個 tool_use 區塊的助理訊息：約 2,000 個 token
- 佔位工具結果：約 500 個 token

共用前綴總計：約 48,500 個 token。各子代理專屬指令：約 200 個 token。

不使用 fork（五個獨立代理，各有新鮮上下文和自己的系統提示詞）：
- 每個子代理處理自己的系統提示詞 + 工具 + 任務提示詞
- 無快取共用（不同的系統提示詞、不同的工具集）
- 成本：5 x 全價輸入處理

使用 fork（位元組相同的前綴）：
- 子代理 1：48,700 個 token 全價（第一次請求，快取未命中）
- 子代理 2-5：48,500 個 token 以 10% 的價格（快取命中）+ 各 200 個 token 全價
- 子代理 2-5 的有效成本：約 4,850 + 200 = 約每個 5,050 個 token 等值

節省效果隨上下文大小和子代理數量成比例放大。對於一個有 100K token 歷史並派生 8 個平行 fork 的暖會話，快取節省可超過無共用時輸入 token 成本的 90%。

這就是為什麼 fork 系統的每個設計決策——傳遞而非重新計算、精確工具傳遞、佔位結果、甚至保留 Agent 工具在子代理池中儘管它被禁止——都在優化同一件事：位元組相同的前綴。每個決策都用少量的優雅性或安全性換來可量化的 API 成本降低。

## 設計張力

Fork 系統做了一些明確的取捨，值得理解：

**隔離性 vs. 快取效率。** Fork 子代理繼承一切，包括與其任務無關的對話歷史。一個重寫測試的子代理不需要父代理討論資料庫 schema 設計的那 15 則訊息。但包含這些訊息才能讓前綴相同。去除無關歷史會節省上下文視窗空間，但代價是破壞快取。這個設計賭的是快取節省優於額外的上下文負擔。

**安全性 vs. 快取效率。** Agent 工具留在 fork 子代理的工具池中，儘管子代理不能使用它。移除它會更安全（子代理甚至無法嘗試 fork），但會改變工具陣列序列化。樣板標籤和遞迴 fork 防護是補償控制措施——執行期預防，而非靜態移除。

**簡潔性 vs. 快取效率。** 佔位工具結果是一種欺騙。無論父代理的助理訊息中那些工具呼叫實際上做了什麼，子代理看到的每個 tool_use 區塊都是 `'Fork started -- processing in background'`。這沒問題，因為子代理的指令告訴它該做什麼——它不需要父代理派發回合中準確的工具結果。但這意味著子代理的對話歷史在技術上是不連貫的。佔位字串的選擇是為了簡潔和統一，而非準確。

這些取捨都反映了同一個優先考量：當你按每個 token 付費進行規模化 API 呼叫時，位元組相同的前綴值得為此調整整個架構。

## 應用這個思路：為提示詞快取效率而設計

Fork 代理模式適用於 Claude Code 以外的場景。任何從相同上下文派發多個平行 LLM 呼叫的系統，都能從快取感知的請求構造中獲益。原則如下：

**1. 傳遞已渲染的提示詞，不要重新計算。** 如果你的系統提示詞包含任何動態內容——功能旗標、時間戳、使用者偏好、A/B 測試變體——就捕獲渲染結果並按值傳遞給子代理。重新計算有分歧的風險。

**2. 凍結工具陣列。** 如果你的子代理需要不同的工具集，你就放棄了工具區塊的快取共用。考慮保留完整工具集，並使用執行期防護（像 fork 樣板的「不要使用 Agent」）而非編譯期移除。

**3. 最大化共用前綴，最小化各子代理後綴。** 構造你的訊息陣列時，讓所有共用內容在前，各子代理內容附加在最後。共用和各子代理內容交錯會碎片化快取邊界。

**4. 對可變內容使用常數佔位符。** 當訊息結構需要對前一個工具呼叫的回應時，在所有子代理中使用相同的佔位字串，而非實際的（會分歧的）結果。

**5. 衡量損益平衡點。** 快取共用有成本：每個子代理更大的上下文視窗（它們攜帶無關的歷史）、執行期防護而非靜態安全、架構複雜性。計算你的平行模式（多少個子代理、共用前綴多大）在考慮額外的上下文 token 後是否真的省錢。

Fork 代理系統，從根本上說，是一個提示詞快取利用引擎。它回答了每個多代理系統建構者最終都會面對的問題：當快取給你重複前綴 90% 的折扣時，你願意為了獲得這個折扣把架構改造到什麼程度？Claude Code 的答案是：非常遠。
