# 第十八章：我們學到了什麼

## 五個架構賭注

Claude Code 不是唯一的代理系統，也不是最早的。但它押下了五個讓它有別於代理框架生態的架構賭注。在將近兩千個檔案與十七個章節之後，這些賭注值得好好審視。

### 賭注一：Generator 循環，而非 Callback

大多數代理框架提供一條流水線：定義工具、註冊處理器、讓框架負責協調。開發者撰寫 callback，框架決定何時呼叫它們。

Claude Code 反其道而行。`query()` 函式是一個非同步 generator——開發者擁有循環的控制權。模型串流回應，generator 產生（yield）工具呼叫，呼叫者執行它們、附加結果，然後 generator 再次循環。只有一個函式、一條資料流、一個所有互動都必須經過的地方。generator 回傳型別的 10 個終止狀態與 7 個延續狀態，編碼了每一種可能的結果。循環就是系統。

這個賭注是：即使一個 generator 函式成長到 1,700 行，它也比分散的 callback 圖更容易理解。研究過原始碼之後，這個賭注得到了回報。當你想了解一個工作階段為何結束，你只需看一個函式。當你想新增一個終止狀態，你只需在一個可辨識的聯合型別上新增一個變體。型別系統強制要求窮舉處理。Callback 架構會將這些邏輯分散到數十個檔案中，而 callback 之間的互動將是隱式的，而非在控制流程中清晰可見的。

### 賭注二：基於檔案的記憶，而非資料庫

第十一章詳細說明了理由，但架構意義不止於記憶。選擇使用純 Markdown 檔案而非 SQLite、向量資料庫或雲端服務，是一個以透明性換取能力的賭注。資料庫能支援更豐富的查詢、更快的查找與事務保證。檔案提供這些都沒有。檔案提供的是信任。

一個使用者用 vim 打開 `~/.claude/projects/myapp/memory/MEMORY.md`，直接看到代理程式記得關於他們的什麼，與必須問代理程式「你記得什麼？」然後希望答案完整的使用者，對系統有著根本上不同的關係。基於檔案的設計讓代理程式的知識狀態可以從外部觀察，而非只能靠自我陳述。這比查詢效能更重要。LLM 驅動的召回系統用檢索智慧彌補了儲存的簡單性——一個 Sonnet 側查詢從清單中選出五個相關記憶，比嵌入向量相似度更精確，且不需要任何基礎設施。

### 賭注三：自我描述的工具，而非中央協調器

代理框架通常提供一個工具註冊表：你在中央設定中描述你的工具，框架再將它們呈現給模型。Claude Code 的工具描述自己。每個 `Tool` 物件攜帶自己的名稱、描述、輸入 schema、提示貢獻、並行安全旗標和執行邏輯。工具系統的工作不是向模型描述工具——而是讓工具描述自己。

這個賭注在可擴展性上得到回報。MCP 工具（第十五章）透過實作同一套介面成為一等公民。來自 MCP 伺服器的工具與內建工具對模型來說無從區分。系統不需要一個單獨的「MCP 工具轉接器」層——封裝產生了一個標準的 `Tool` 物件，從那時起，現有的工具流水線就會處理它：權限檢查、並行執行、結果預算、鉤子攔截。

### 賭注四：Fork 代理以共享快取

第九章介紹了 fork 機制：一個子代理在其上下文視窗中攜帶父代理的完整對話，共享父代理的提示快取。這不是一個便利性優化——這是一個認為快取共享模式值得 fork 生命週期管理複雜性的架構賭注。

替代方案——產生一個帶有對話摘要的全新代理——更簡單但代價高昂。每個全新代理都要付出從頭處理上下文的全部成本。Fork 出的代理免費獲得父代理的已快取前綴（輸入 token 九折優惠），使得為小任務產生代理在經濟上可行：記憶提取、程式碼審查、驗證流程。背景記憶提取代理（第十一章）在每次查詢循環後執行，其成本之所以微不足道，正是因為它共享父代理的快取。沒有基於 fork 的快取共享，那個代理的成本就會高到令人卻步。

### 賭注五：鉤子，而非外掛程式

大多數可擴展性系統使用外掛程式——在宿主行程內執行並註冊功能的程式碼。Claude Code 使用鉤子——在生命週期節點執行的外部行程，透過退出碼和 stdin/stdout 上的 JSON 溝通。

這個賭注是：行程隔離值得那個開銷。外掛程式可能使宿主崩潰；鉤子只會使自己的行程崩潰。外掛程式可能洩漏記憶體進入宿主的堆積；鉤子的記憶體隨其行程消亡。外掛程式需要一個必須被版本化和維護的 API 表面；鉤子只需要 stdin、stdout 和一個退出碼——一個自 1971 年以來就穩定的協定。

開銷是真實的：每次鉤子呼叫產生一個行程，要花費幾毫秒，而同行程的 callback 不需要這個成本。內部 callback 的 -70% 快速路徑（第十二章）表明系統清楚這個成本的重要性。但對於外部鉤子——使用者腳本、團隊 linter、企業政策伺服器——隔離保證讓系統更安全地擴展。企業可以部署基於鉤子的政策執行，而不必擔心一個格式錯誤的鉤子腳本會使開發者的工作階段崩潰。

## 哪些可以移植，哪些不行

並非 Claude Code 的每個模式都能泛化。有些是規模、資源或特定約束的產物，其他代理開發者可能不會面臨這些。

### 可以移植到任何代理的模式

**Generator 循環模式。** 任何需要串流回應、處理工具呼叫並管理多個終止狀態的代理，都能從讓循環顯式化（而非隱藏在 callback 後面）中受益。可辨識的聯合型別回傳值——精確編碼循環停止原因——是一個消除整個類別「代理為何停止？」除錯問題的模式。

**基於檔案的記憶與 LLM 召回。** 具體實作細節屬於 Claude Code，但原則——簡單儲存結合智慧檢索——適用於任何需要跨工作階段持久化知識的代理。四類分類法（user、feedback、project、reference）與可導出性測試（「這能從目前的專案狀態重新導出嗎？」）是可重用的設計啟發法。

**非對稱讀/寫通道用於遠端執行。** 當讀取是高頻串流而寫入是低頻 RPC 時，分離它們是正確的，無論具體的傳輸協定是什麼。

**點陣圖預過濾用於搜尋。** 任何搜尋大型檔案索引的代理都能從 26 位元字母點陣圖預過濾中受益。每條目 4 位元組，每個候選者一次整數比較——成本效益比非常出色。

**提示快取穩定性作為架構考量。** 如果你的代理使用有提示快取的 API，將提示結構化為穩定內容在前、易變內容在後，這不是優化——這是決定你成本結構的架構決策。

### 特定於 Claude Code 規模的模式

**Fork 出的終端機渲染器。** Claude Code fork 了 Ink 並用壓縮型別化陣列、基於物件池的字符駐留與儲存格層級差異比對重新實作了渲染流水線，因為它需要在終端機中 60fps 串流。大多數代理渲染到網頁介面或簡單的日誌輸出。只有當終端機渲染是你的主要 UI 且你以高頻率串流時，這項工程投資才有意義。

**超過 50 個啟動分析檢查點。** 當你有數十萬使用者、0.5% 採樣能產生具統計意義的數據時，這才有意義。對於較小的代理，更簡單的計時系統就夠了。

**八種 MCP 傳輸類型。** Claude Code 支援 stdio、SSE、HTTP、WebSocket、SDK、兩種 IDE 變體和一個 Claude.ai 代理，因為它必須整合每一種部署拓撲。大多數代理只需要 stdio 和 HTTP。

**鉤子快照安全模型。** 在啟動時凍結鉤子設定並永遠不隱式重新讀取，是針對一個特定威脅的防禦：惡意倉庫程式碼在使用者接受信任對話後修改鉤子。當你的代理在帶有不受信任 `.claude/` 設定的任意倉庫中執行時，這才重要。只在受信任環境中執行的代理可以使用更簡單的鉤子管理。

## 複雜性的代價

將近兩千個檔案。這買到了什麼，代價又是什麼？

檔案數量作為複雜性指標是有誤導性的。其中很多是測試基礎設施、型別定義、設定 schema，以及 fork 出的 Ink 渲染器。實際的行為複雜性集中在少數幾個高密度檔案中：`query.ts`（1,700 行，代理循環）、`hooks.ts`（4,900 行，生命週期攔截系統）、`REPL.tsx`（5,000 行，互動協調器），以及記憶系統的提示建構函式。

複雜性來自三個來源，各有不同的特質：

**協定多樣性。** 支援五種終端機鍵盤協定、八種 MCP 傳輸類型、四種遠端執行拓撲和七個設定範疇，本就是複雜的。每增加一個協定是程式碼庫的線性增長，而非指數增長——但總和很大。這種複雜性在 Brooks 的意義上是偶然的：它來自環境（終端機碎片化、MCP 傳輸演進、遠端部署拓撲），而非被解決的問題本身。

**效能優化。** 基於物件池的渲染、點陣圖搜尋預過濾、黏性快取鎖存和推測性工具執行，每一個都以增加複雜性換取可量測的效能提升。這種複雜性是有依據的——每項優化都先有分析數據指出瓶頸。風險在於優化不斷積累並以使熱路徑更難修改的方式相互作用。

**行為調校。** 記憶系統的提示指令、過期警告、驗證協定、「忽略記憶」反模式指令——這些不是程式碼複雜性。它們是提示複雜性，帶有不同的維護負擔。當模型在不同版本間的行為改變時，透過評估（eval）精心調校的提示指令可能需要重新調校。貫穿程式碼庫的評估基礎設施（以案例編號和評估分數引用）是防止退化的防線，但需要持續投入。

這個系統的維護負擔是重大的。新工程師閱讀程式碼庫時，不僅要理解程式碼路徑，還要理解促使特定提示措辭的評估結果、促使特定安全檢查的生產事故，以及促使特定優化的效能剖析。程式碼註解很詳盡——許多包含評估案例編號和前後量測——但將近兩千個檔案中的詳盡註解本身就是一種閱讀負擔。

## 代理系統的走向

從 Claude Code 的模式中可以看到四個趨勢，指向這個領域的發展方向。

### MCP 作為通用協定

第十五章描述 Claude Code 是最完整的 MCP 用戶端之一。其意義不在於 Claude Code 的實作——而在於 MCP 的存在本身。一個標準化的工具探索與呼叫協定意味著為一個代理構建的工具可以與任何代理配合使用。生態系統效應是顯而易見的：一旦為 Postgres 構建了 MCP 伺服器，就能服務每一個說 MCP 的代理。開發者在工具整合上的投資是可移植的。

對代理開發者的啟示：如果你在定義一個自訂工具協定，你很可能在犯一個錯誤。MCP 已經足夠好，而且在持續改善，標準協定的生態優勢隨時間複利增長。構建一個 MCP 用戶端，為規範做貢獻，讓協定透過社群回饋演進。

### 多代理協調

Claude Code 的子代理系統（第八章）、任務協調（第十章）和 fork 機制（第九章）是多代理模式的早期實作。它們解決了特定問題——快取共享、並行探索、結構化驗證——但也揭示了根本挑戰：協調開銷。

代理間的每條訊息都消耗 token。每次 fork 共享快取，但增加了父代理最終必須協調的對話分支。Task 系統的狀態機（queued、running、completed、failed、cancelled）是增加複雜性但不增加能力的協調機制。隨著代理能力增強，壓力將從「如何協調多個代理？」轉向「如何讓一個代理足夠強大，使協調變得不必要？」

目前的證據表明兩種方法將並存。簡單任務將使用單一代理。複雜任務將使用協調的多代理系統。工程挑戰在於讓協調開銷低到足以讓多代理在真正並行的工作中有利，而不僅僅是對複雜的工作有利。

### 持久記憶

Claude Code 的記憶系統是持久代理記憶的第一版。基於檔案的設計、四類分類法、LLM 驅動的召回、過期系統，以及長期執行工作階段的 KAIROS 模式，都是對一個將大幅演進的問題的第一代解決方案。

未來的記憶系統可能會增加結構化檢索（目前系統檢索整個檔案；未來系統可能檢索特定事實）、跨專案遷移學習（到處適用的使用者偏好、不跨出的專案慣例）以及協作記憶（第十一章的團隊記憶是第一步，但同步、衝突解決和存取控制都很簡單）。

開放問題在於基於檔案的方法是否能擴展。在每個專案 200 個記憶的規模下，它可行。在每個專案 2,000 個記憶的規模下，Sonnet 側查詢的清單會變得太大、整合會變得太昂貴、索引會超過其上限。隨著使用增長，「檔案優於資料庫」的架構賭注將面臨最嚴峻的考驗。

### 自主運作

KAIROS 模式、背景記憶提取代理、自動整合夢境、推測性工具執行——這些都是邁向自主運作的步驟。代理在未被要求的情況下做有用的事：它記住了你忘記告訴它要記住的，在你睡覺時整合自己的知識，在目前的回應完成前就開始執行下一個工具。

軌跡很清晰。未來的代理將更少是被動回應，更多是主動出擊。它們會注意到使用者沒有描述的模式，提出使用者沒有請求的修正，在沒有明確 `/remember` 指令的情況下維護自己的知識。Claude Code 的記憶系統，憑藉其背景提取安全網和精心設計的「應該保存什麼」啟發法，是這個未來的原型。

約束是信任。自主運作需要使用者相信代理在無人看管時會做正確的事。基於檔案的記憶、可觀察的鉤子系統、過期警告、權限對話——所有這些都存在，因為信任必須被賺取，而非假設。通往更自主代理的路，穿越的是更透明的代理。

## 結語

十七個章節。六個核心抽象。一個 generator 循環居於中心，工具向外延伸，記憶向過去伸展，鉤子守護邊界，一個渲染引擎將一切轉化為螢幕上的字元，而 MCP 將其連接到程式碼庫之外的世界。

Claude Code 最深層的模式不是任何單一技術。而是一個反覆出現的決定：將複雜性推向邊界。渲染系統將複雜性推向物件池和差異比對——在流水線內部，一切都是整數比較。輸入系統將複雜性推向分詞器和按鍵綁定解析器——在處理器內部，一切都是型別化的動作。記憶系統將複雜性推向寫入協定和召回選擇器——在對話內部，一切都是上下文。代理循環將複雜性推向終止狀態和工具系統——在循環內部，就只是：串流、收集、執行、附加、重複。

每個邊界吸收混亂並輸出秩序。原始位元組變成 `ParsedKey`。Markdown 檔案變成召回的記憶。MCP JSON-RPC 變成 `Tool` 物件。鉤子退出碼變成權限決策。在每個邊界的一側，世界是混亂的——五種鍵盤協定、脆弱的 OAuth 伺服器、過期的記憶、不受信任的倉庫鉤子。在另一側，世界是有型別的、有界的、被窮舉處理的。

如果你正在構建一個代理系統，這就是可移植的教訓。不是具體的技術——你可能不需要基於物件池的渲染、KAIROS 模式或八種 MCP 傳輸。而是這個原則：定義你的邊界，在那裡吸收複雜性，讓邊界之間的一切保持清晰。邊界是工程困難的地方。內部是工程愉快的地方。為愉快的內部而設計，並將你的複雜性預算投資在邊緣。

原始碼是開放的。螃蟹爪中握著地圖。去讀吧。
